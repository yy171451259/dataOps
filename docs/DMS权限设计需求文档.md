# 数据管理平台 — 权限控制设计需求文档

> 参考标准：阿里云 DMS 细粒度权限管理  
> 对标版本：安全协同模式（Security Collaboration Mode）

---

## 一、权限体系总览

### 1.1 权限模型

采用 **双轨权限模型**：

| 模型 | 描述 | 对标 DMS |
|------|------|----------|
| **基于资源的授权（RBAC + ACL）** | 将资源（实例/库/表/列）作为 Owner 分配给用户，Owner 可二次授权 | 实例 Owner / 库 Owner / 表 Owner |
| **基于操作的授权（Permission-based）** | 预定义操作权限码，通过角色或直接授予用户 | 普通权限（查询/变更/导出） |

校验优先级：**DMS 内部权限 > 外部角色权限**，管理员可配置关闭外部权限校验，强制仅使用 DMS 内部权限体系。

### 1.2 核心概念

```
用户(User) ──多对多──▶ 角色(Role) ──多对多──▶ 权限码(Permission)
   │                                              │
   │                   直接授权                     │
   └────────────────────────────────────────────────┘
   │
   └── Owner 关系 ──▶ 资源(实例/库/表)
                           │
                      可二次授权给其他用户
```

## 二、权限层级（五级资源粒度）

对标 DMS 五级权限粒度，从粗到细：

| 层级 | 资源对象 | 权限粒度 | 前提条件 |
|------|----------|----------|----------|
| **L1 实例级** | 数据库实例 | 实例下所有库的查询/导出/变更权限（不含敏感列和行管控数据） | — |
| **L2 库级** | 单个Schema | 指定Schema下所有表的查询/导出/变更权限 | — |
| **L3 表级** | 单张数据表 | 指定表的查询/导出/变更权限 | — |
| **L4 敏感列级** | 标记为敏感的字段 | 敏感字段的查询/导出/变更 | 需开通敏感数据保护 + 拥有对应库/表权限 |
| **L5 行级** | 满足行级管控规则的数据行 | 受限行数据的查询/导出/变更 | 需拥有对应库/表权限 |
| **扩展** | 可编程对象 | 存储过程、函数、触发器 | 仅安全托管实例 |

### 2.1 权限继承规则

```
实例权限 ⊃ 库权限 ⊃ 表权限 ⊃ 列权限
```

- 拥有实例权限，自动覆盖该实例下所有库/表/列（敏感列除外）
- 但 **敏感列权限** 和 **行级权限** 不继承，必须单独申请

## 三、权限类型定义

### 3.1 操作权限（Permission Codes）

对标 DMS 的核心操作权限，对应的权限码体系：

#### 3.1.1 SQL 操作类

| 权限码 | 操作 | 说明 | 风险等级 |
|--------|------|------|----------|
| `sql:query` | 数据查询 | 执行 SELECT / SHOW / DESCRIBE / EXPLAIN 等只读 SQL | 低 |
| `sql:update` | 数据变更 | 执行 INSERT / UPDATE / DELETE 等 DML 操作 | 中 |
| `sql:ddl` | 结构变更 | 执行 ALTER / CREATE / DROP / TRUNCATE 等 DDL 操作 | 高 |
| `sql:export` | 数据导出 | 导出查询结果为文件（CSV/Excel/SQL） | 中 |
| `sql:audit` | SQL 审核 | 审核 SQL 语句规范性、性能风险 | 低 |

#### 3.1.2 工单管理类

| 权限码 | 操作 | 说明 |
|--------|------|------|
| `ticket:create` | 创建工单 | 提交数据变更/导出/结构变更工单 |
| `ticket:approve` | 审批工单 | 审批/拒绝待处理的工单，控制 DML 暂停/继续/终止 |
| `ticket:rollback` | 回滚工单 | 回滚已执行完成的变更工单 |
| `ticket:cancel` | 取消工单 | 取消自己创建的待审批工单 |

#### 3.1.3 数据库管理类

| 权限码 | 操作 | 说明 |
|--------|------|------|
| `schema:view` | 查看Schema | 查看实例列表、Schema列表 |
| `schema:manage` | 管理Schema | 添加/删除/修改实例配置 |
| `metadata:view` | 查看元数据 | 查看表结构、索引、外键等信息 |
| `metadata:manage` | 管理元数据 | 修改表结构、创建/删除索引等 |

#### 3.1.4 安全管理类

| 权限码 | 操作 | 说明 |
|--------|------|------|
| `user:manage` | 用户管理 | 创建/修改/禁用用户，管理用户权限 |
| `role:manage` | 角色管理 | 创建/修改/删除角色，分配权限 |
| `permission:manage` | 权限管理 | 管理权限码体系 |
| `audit:view` | 查看审计日志 | 查看操作审计记录 |
| `masking:manage` | 脱敏管理 | 配置数据脱敏规则 |
| `system:settings` | 系统设置 | 管理系统级配置参数 |

#### 3.1.5 监控与导入导出类

| 权限码 | 操作 | 说明 |
|--------|------|------|
| `monitor:view` | 查看监控 | 查看实例性能监控面板 |
| `import:execute` | 数据导入 | 执行数据导入任务 |
| `export:data` | 数据导出 | 提交数据导出工单 |
| `ddl:workbench` | DDL 工作台 | 使用 DDL 可视化设计器 |
| `pipeline:manage` | DDL 流水线 | 管理 DDL 发布流水线 |

### 3.2 数据权限（Owner 权限）

Owner 是一种**资源级管理权限**，不依赖权限码体系：

| Owner 类型 | 权限范围 | 特有操作 |
|------------|----------|----------|
| **实例 Owner** | 整个数据库实例 | 可查看/授予/回收该实例的权限，可查询实例下所有数据（不含敏感列） |
| **库 Owner** | 单个数据库 | 可查看/授予/回收该库的权限 |
| **表 Owner** | 单张数据表 | 可查看/授予/回收该表的权限 |

**Owner 继承**：实例 Owner 自动成为该实例下所有库的 Owner，库 Owner 自动成为该库下所有表的 Owner。

## 四、安全规则与访问控制

### 4.1 安全模式

| 模式 | 说明 | 权限粒度 |
|------|------|----------|
| **标准模式** | 仅需数据库账号密码即可登录查询 | 粗粒度（仅库级） |
| **安全协同模式（推荐）** | 必须通过权限申请/审批才能访问数据 | 五级细粒度 |

### 4.2 元数据访问控制

控制用户对实例/数据库/表的**可见性**：

| 控制类型 | 规则 | 开启后效果 |
|----------|------|------------|
| **实例访问控制** | 仅已授权用户可搜索和访问该实例 | 未授权用户无法在对象浏览器中看到该实例，无法主动发起权限申请 |
| **数据库访问控制** | 仅已授权用户可搜索和访问该库 | 未授权用户看不到该库 |
| **用户访问控制** | 用户只能看到已获得授权的资源 | 用户无法搜索到未授权的任何资源 |

> 判断 "已授权" 的标准：拥有上述任意一种操作权限或数据权限（Owner）。

### 4.3 敏感数据保护

| 功能 | 说明 | 前置条件 |
|------|------|----------|
| **敏感列标记** | 对表中包含敏感数据的字段（手机号、身份证等）打标 | 已开通敏感数据保护 |
| **敏感列权限** | 单独申请敏感列的查询/导出/变更权限 | 已开通敏感数据保护 + 拥有对应库/表权限 |
| **数据脱敏** | 对敏感字段按规则自动脱敏（如 `138****1234`） | 配置脱敏规则 |

### 4.4 行级管控

限制用户只能查看/操作符合特定条件的**数据行**：

```
示例：开发者A只能查看 department='研发部' 的数据行
```

- 需先拥有对应库/表权限作为前提
- 可在权限申请时指定行过滤条件

## 五、审批流程

### 5.1 差异化审批策略

支持按以下维度配置不同的审批链路：

| 维度 | 场景示例 | 审批策略 |
|------|----------|----------|
| **按环境** | 生产环境 vs 测试环境 | 生产：三级审批（项目经理→DBA→安全官）；测试：一级审批或无审批 |
| **按风险等级** | 大表变更 > 1万行 | 自动上升审批级别 |
| **按资源敏感度** | 核心业务表 vs 普通表 | 敏感资源：严格审批 |
| **按时间窗口** | 非工作时间执行变更 | 额外增加审批节点 |

### 5.2 审批节点

| 角色 | 审批职责 |
|------|----------|
| **项目经理/Leader** | 确认业务合理性 |
| **DBA** | 审核 SQL 规范、性能影响、锁风险 |
| **安全管理员** | 审核敏感数据操作合规性 |
| **实例 Owner** | 确认对自身负责资源的操作授权 |

### 5.3 审批状态机

```
[申请提交] → [待审批] → [项目经理审批] → [DBA审批] → [执行]
                ↓                ↓
           [已拒绝]         [已拒绝]
```

## 六、用户权限生命周期

### 6.1 权限申请流程

```
┌─────────┐     ┌─────────────┐     ┌──────────────┐     ┌──────────┐
│ 用户发现  │ →  │ 搜索目标资源  │ →  │ 申请对应权限   │ →  │ 审批通过  │ →  │ 生效 │
│ 需要权限  │     │ (在访问控制内) │     │ (选择权限类型) │     │           │     │      │
└─────────┘     └─────────────┘     └──────────────┘     └──────────┘     └──────┘
                      │
                      └── 若资源开启访问控制且用户无任何权限 → 搜索不到，需管理员手动添加
```

### 6.2 权限回收

- **管理员主动回收**：管理员/DBA 可随时回收任意用户的权限
- **Owner 回收**：资源 Owner 可回收自己授予的权限
- **定期自动回收**：超出有效期（如 30 天）自动回收，需重新申请
- **离职自动回收**：用户被禁用/删除时自动回收所有权限

### 6.3 查看我的权限

用户可查看自己拥有的：
- 操作权限列表（按权限码展示）
- Owner 关系列表（按资源层级展示）
- 权限有效期和来源（直接授予 / 角色继承）

## 七、技术实现要点

### 7.1 权限校验流程

每次数据库操作前，系统按以下顺序校验：

```
1. 用户是否已登录？（JWT 认证，401）
2. 用户是否拥有对应的权限码？（Spring Security @PreAuthorize，403）
3. 目标资源是否开启了访问控制？
4. 权限是否覆盖目标资源层级？（实例/库/表/列/行）
5. 操作是否在权限类型范围内？（查询/变更/导出/DDL）
```

### 7.2 数据模型设计

#### 核心表结构

```sql
-- 用户表
users (id, username, password_hash, nickname, status, is_admin, created_at)

-- 角色表
roles (id, name, code, description, created_at)

-- 权限码表
permissions (id, code, name, category, description)

-- 角色-权限关联
role_permissions (role_id, permission_code)

-- 用户-角色关联
user_roles (user_id, role_id, grantor_id, granted_at, expired_at)

-- 用户-权限直接关联（绕过角色）
user_permissions (user_id, permission_code, grantor_id, granted_at, expired_at)

-- 资源 Owner 关系
resource_owners (resource_type, resource_id, owner_user_id, created_at)

-- 元数据访问控制
metadata_access_controls (resource_type, resource_id, enabled, created_at)

-- 用户权限申请记录
permission_requests (id, user_id, resource_type, resource_id, 
                     requested_permissions, status, reason, 
                     created_at, approved_at)

-- 敏感列标记
sensitive_columns (table_id, column_name, sensitivity_level, mask_rule)

-- 行级管控规则
row_level_rules (table_id, rule_name, filter_condition, target_user_id)
```

### 7.3 与现有系统的集成

当前项目已有基础 RBAC 体系（`SecurityConfig` + `JwtAuthFilter` + `PermissionMapper`），需要扩展：

| 现有能力 | 需扩展 |
|----------|--------|
| JWT 认证 + Token 刷新 | ✅ 已实现 |
| 权限码校验 (`hasAuthority`) | ✅ 已实现 |
| 角色-权限关联 | ✅ 已实现 |
| 用户-角色关联 | ✅ 已实现 |
| **资源级 Owner** | ❌ 需新增 |
| **元数据访问控制** | ❌ 需新增 |
| **敏感列保护 + 脱敏** | ❌ 需新增 |
| **行级管控** | ❌ 需新增 |
| **权限有效期 + 自动回收** | ❌ 需新增 |
| **权限申请 + 审批流** | ❌ 需新增 |
| **差异化审批流程** | ❌ 需新增 |
| **可编程对象权限** | ❌ 需新增 |

## 八、实施优先级建议

| 阶段 | 功能模块 | 优先级 | 预计工作量 |
|------|----------|--------|------------|
| **P0 基础** | 完善现有 RBAC 权限码体系，补齐缺失的权限码 | 高 | 2 天 |
| **P1 资源** | 实现资源 Owner 关系（实例/库/表级） | 高 | 3 天 |
| **P1 元数据** | 实现元数据访问控制（资源搜索可见性） | 高 | 2 天 |
| **P1 敏感** | 敏感列标记 + 脱敏规则配置 + 查询自动脱敏 | 中 | 4 天 |
| **P2 审批** | 差异化审批流程引擎 | 中 | 5 天 |
| **P2 权限申请** | 用户自主权限申请 + 管理员授权界面 | 中 | 3 天 |
| **P2 行级** | 行级管控规则 | 低 | 3 天 |
| **P3 可编程对象** | 存储过程/函数权限 | 低 | 2 天 |
| **P3 自动回收** | 权限有效期 + 定时回收 | 低 | 1 天 |
